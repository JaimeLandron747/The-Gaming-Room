# The-Gaming-Room
Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?
--The client was The Gaming Room, a company seeking to expand their game Draw It or Lose It into a web-based, cross-platform application. The software needed to support multiple games, teams, and players while ensuring unique names and identifiers. The design focused on scalability, centralized management, and browser accessibility.

What did you do particularly well in developing this documentation?
--I organized the system architecture and domain model clearly to show how the Game, Team, and Player classes interact. The UML diagram and design explanations helped communicate how the Singleton pattern supports centralized control and uniqueness requirements.


What about the process of working through a design document did you find helpful when developing the code?
--The design document helped define requirements and structure before coding began. Planning class relationships and constraints early reduced confusion and made implementation more organized.

If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
--I would improve the platform evaluation section by adding more detailed comparisons related to cloud deployment and long-term maintenance. Including more real-world examples would strengthen the recommendations.

How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?
--I focused on scalability, accessibility, and preventing duplicate data to meet the client’s requirements. Considering user needs ensures the system remains reliable and functional under real-world usage.

How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?
--I approached design by analyzing requirements first and then creating an architecture supported by object-oriented principles and UML modeling. In the future, I would continue using structured planning and incorporate earlier testing and feedback.
